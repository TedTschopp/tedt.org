# Commit Message Guidelines

Use **Conventional Commits** style for clarity and automation compatibility.

### Format
```

<type>(scope): short description

```

### Allowed types
- **feat** â€“ new feature (e.g., new layout, include, or Liquid tag)
- **fix** â€“ bug fix (e.g., broken link, layout rendering issue)
- **style** â€“ formatting, CSS updates, or non-functional visual tweaks
- **content** â€“ new or updated text, Markdown, or data
- **refactor** â€“ code restructuring without functional change
- **build** â€“ changes to build or deployment (e.g., `_config.yml`)
- **chore** â€“ maintenance, dependency bumps, or meta changes
- **docs** â€“ documentation-only updates

### Examples
```

feat(layout): add dark mode toggle to header
fix(js): resolve broken event listener in search script
content(blog): publish new post on folklore and myth
style(css): improve spacing in article sections
build(config): enable pagination plugin in Jekyll

```

### Best practices
- Keep the subject line under **72 characters**.
- Use the **imperative mood** ("add" not "added").
- Reference issues when relevant: `fixes #123`.
- Make commits small and atomic (one concern per commit).
